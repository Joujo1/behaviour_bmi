## 1. Database Schema

### Tables and columns

There are six tables. All primary keys are auto-incrementing integers (`SERIAL`).

---

**`training_stages`**
Top-level groupings of the curriculum, e.g. "Habituation" or "Easy clicks".

- `id` ‚Äî primary key
- `name` ‚Äî unique text label shown in the UI
- `description` ‚Äî optional free text
- `sort_order` ‚Äî integer controlling display order in the curriculum editor sidebar

---

**`training_substages`**
One row per curriculum level. This is the central definition table ‚Äî it carries both the trial definition (what the Pi runs) and the rules for when to move to the next level.

- `id` ‚Äî primary key
- `stage_id` ‚Äî foreign key ‚Üí `training_stages.id`
- `substage_number` ‚Äî integer ordering within the parent stage
- `label` ‚Äî display name, e.g. "Easy left-biased"
- `task_config` ‚Äî JSONB: the complete trial definition sent to the Pi (states, transitions, actions). Same format as what the curriculum editor builds.
- `advance_criteria` ‚Äî JSONB: rules for moving forward. Null means manual-only advancement. Format: `{"type": "pct_correct", "window": 20, "threshold": 0.80}`
- `fallback_criteria` ‚Äî JSONB: rules for moving backward. Same format. Null means no automatic fallback.
- `advance_to_substage_id` ‚Äî foreign key ‚Üí `training_substages.id` (self-referential). Null means end of curriculum.
- `fallback_to_substage_id` ‚Äî foreign key ‚Üí `training_substages.id` (self-referential). Null means no fallback defined.
- `retired` ‚Äî boolean: if true, the substage is hidden from all UI dropdowns but not deleted

---

**`subjects`**
One row per animal.

- `id` ‚Äî primary key
- `code` ‚Äî unique text identifier, e.g. "R001"
- `sex` ‚Äî single character: "M" or "F"
- `dob` ‚Äî date of birth
- `weight_g` ‚Äî body weight in grams
- `water_restricted` ‚Äî boolean
- `enrolled_at` ‚Äî timestamp when the animal was registered
- `current_substage_id` ‚Äî foreign key ‚Üí `training_substages.id`. This is updated automatically by the advancement evaluator after each trial, or manually by the researcher via the subjects page.
- `notes` ‚Äî free text

---

**`sessions`**
One row per experimental sitting ‚Äî one animal in one cage for one day's work.

- `id` ‚Äî primary key
- `researcher` ‚Äî name of who ran the session
- `notes` ‚Äî free text
- `subject_id` ‚Äî foreign key ‚Üí `subjects.id`
- `substage_id` ‚Äî foreign key ‚Üí `training_substages.id`. Snapshotted from `subjects.current_substage_id` at the moment the session is opened. This means the session record stays correct even if the animal advances to a new substage mid-session.
- `weight_g` ‚Äî animal weight recorded at session start or close
- `water_ml` ‚Äî water given during the session
- `started_at` ‚Äî timestamp when the session was opened
- `closed_at` ‚Äî timestamp when the session was closed. Null means the session is still open.

---

**`trial_results`**
One row per trial completion or abort received from the Pi.

- `id` ‚Äî primary key
- `cage_id` ‚Äî integer: physical cage number (1‚Äì12)
- `trial_id` ‚Äî text UUID generated by the Pi engine for this trial
- `outcome` ‚Äî one of: `correct`, `wrong`, `aborted`
- `events` ‚Äî JSONB array: the full timestamped event log from the Pi for this trial. Each entry has a time offset `t` in seconds from trial start, plus either a state transition (`from`/`to`) or a sensor event (`sensor`, `active`).
- `session_id` ‚Äî foreign key ‚Üí `sessions.id`. Null for one-shot dev runs that bypass the session workflow.
- `substage_id` ‚Äî foreign key ‚Üí `training_substages.id`. Null for one-shot dev runs. Used by the advancement evaluator to count only trials on the current substage.
- `completed_at` ‚Äî timestamp when the result was received and written

---

**`recordings`**
Written by the frame writer in chunks of N frames (configured in `config.DB_CHUNK_SIZE`). Each row covers one chunk of a binary recording file for one cage. Chunked writes make the system crash-safe ‚Äî partial recordings are still usable.

- `id` ‚Äî bigserial primary key
- `cage_id` ‚Äî physical cage number
- `chunk_start_frame` / `chunk_end_frame` ‚Äî Pi sequence numbers for the first and last frame in this chunk
- `chunk_start_ts` / `chunk_end_ts` ‚Äî microsecond timestamps from the Pi hardware clock
- `chunk_byte_offset` ‚Äî byte position of this chunk's start in the binary file
- `chunk_frame_count` ‚Äî number of frames in this chunk
- `created_at` ‚Äî timestamp when this row was written

---

### Foreign key relationships

- `training_substages.stage_id` ‚Üí `training_stages.id` (many substages belong to one stage)
- `training_substages.advance_to_substage_id` ‚Üí `training_substages.id` (self-referential: points to the next level)
- `training_substages.fallback_to_substage_id` ‚Üí `training_substages.id` (self-referential: points to the previous level)
- `subjects.current_substage_id` ‚Üí `training_substages.id` (each animal is on exactly one substage at a time)
- `sessions.subject_id` ‚Üí `subjects.id` (each session belongs to one animal)
- `sessions.substage_id` ‚Üí `training_substages.id` (snapshotted at open time)
- `trial_results.session_id` ‚Üí `sessions.id` (each trial belongs to one session)
- `trial_results.substage_id` ‚Üí `training_substages.id` (each trial is tagged with the substage it was run on)

---

### Key design decisions

| Decision | Reason |
|---|---|
| `sessions.substage_id` is snapshotted at open time | Animal may advance mid-session; the session record stays accurate regardless |
| `trial_results.substage_id` is stamped per-trial | Advancement evaluator queries `WHERE substage_id = X` ‚Äî only counts trials on the current substage, never trials from before the animal was on this level |
| `task_config` lives directly on `training_substages` | A substage IS a trial definition plus curriculum metadata; splitting them into separate tables adds complexity for no benefit |
| `advance_criteria` and `fallback_criteria` are JSONB | New criteria types can be added without schema changes |
| `recordings` tracks chunks not whole files | Crash-safe; partial recordings are still usable for analysis |

---

### Criteria JSONB format

```json
{
  "type":      "pct_correct",
  "window":    20,
  "threshold": 0.80
}
```

- `type` ‚Äî criteria function to apply. Currently only `pct_correct` is implemented.
- `window` ‚Äî how many recent trials to look back. Aborted trials are excluded and do not count toward the window.
- `threshold` ‚Äî fraction of correct outcomes required, as a decimal between 0 and 1. In the UI this is entered as a percentage (0‚Äì100).

---

### `task_config` / trial definition format

This JSONB field is sent verbatim to the Raspberry Pi engine (after click arrays are expanded at runtime). Example:

```json
{
  "trial_id":      "trial_001",
  "initial_state": "cue_center",
  "states": [
    {
      "id":       "cue_center",
      "duration": 10.0,
      "entry_actions": [
        { "type": "led_on",      "target": "center" },
        { "type": "play_clicks", "left_rate": 39, "right_rate": 9, "click_duration": 1.0 }
      ],
      "exit_actions": [
        { "type": "led_off", "target": "center" }
      ],
      "transitions": [
        { "trigger": "beam_break", "target": "center", "hold_ms": 50, "next_state": "reward" },
        { "trigger": "timeout",                                        "next_state": "__wrong__" }
      ]
    }
  ]
}
```

Terminal state names with special meaning: `__correct__` (trial ends as correct), `__wrong__` (trial ends as wrong), `__end__` (trial ends as correct, legacy alias).

## 2. Trial Data Flow

This describes what happens step by step from the researcher clicking **Trial ‚ñ∂** through to automatic advancement firing.

### Preconditions
- A session has been opened for an animal (`POST /session/open`), which has recorded the animal's current substage.
- The dashboard has the session ID stored in `currentSessionId`.
- The runner dropdown is pre-populated with the animal's current substage.

---

### Step 1 ‚Äî Researcher triggers the run

The researcher clicks **Trial ‚ñ∂** on a cage card in the dashboard. The browser sends `POST /cage/N/trial/run` to Flask with:
- `substage_id` ‚Äî which substage to run (pre-selected from the animal's current substage)
- `session_id` ‚Äî the open session
- `base_iti_s` ‚Äî inter-trial interval after a correct outcome
- `fail_iti_s` ‚Äî inter-trial interval after a wrong or aborted outcome

---

### Step 2 ‚Äî Flask loads the trial definition

Flask queries `training_substages` for the row with the given `substage_id` and reads `task_config`. This is the full trial JSON (states, transitions, actions). It then calls `start_run()` in `runner.py`, passing the trial definition, sender object, ITI values, session ID and substage ID. A background runner thread is started for the cage and Flask returns `{ok: true}` to the browser immediately.

---

### Step 3 ‚Äî Runner loop begins

The runner thread enters a `while True` loop. At the top of each iteration:

1. The trial definition is deep-copied and `_expand_clicks()` is called. Any `play_clicks` action that stores rate parameters (`left_rate`, `right_rate`, `click_duration`) has those replaced with pre-generated Poisson click time arrays. A new random seed is used each time, so every trial has a unique click pattern.

2. The expanded trial JSON is serialised and sent to the Pi over TCP as a newline-terminated message.

3. The Pi responds with `ACK:ok` to confirm receipt.

---

### Step 4 ‚Äî Pi runs the state machine

The Pi's `Engine` receives the trial JSON, loads the state machine, and enters the initial state. It drives hardware in response to events:
- Entry actions execute immediately on state entry (LEDs on, valves open, click playback starts)
- GPIO interrupts from beam sensors trigger beam_break transitions
- Timers fire timeout transitions when a state's duration expires
- The click watcher thread fires `clicks_done` transitions when both click trains finish naturally

The engine records every state transition and sensor event with a timestamp offset from trial start.

---

### Step 5 ‚Äî Trial result returns to Flask

When the trial ends (either through a terminal state or the watchdog timer), the Pi's `on_trial_complete` callback sends a JSON event back to the server over TCP:

```json
{
  "event":    "trial_complete",
  "trial_id": "abc123",
  "outcome":  "correct",
  "events":   [{"t": 0.45, "from": "cue_center", "to": "reward"}, ...]
}
```

This arrives at `handle_trial_event()` in `trial.py`, which is called from the TCP reader thread.

---

### Step 6 ‚Äî Result is processed

`handle_trial_event` does four things in order:

1. Calls `on_trial_complete()` in the runner ‚Äî this sets the threading event, which unblocks the runner thread from its `event.wait()`. The runner is now free to start its ITI wait.

2. Reads `session_id` and `substage_id` from `get_run_context()` ‚Äî these were stored when `start_run()` was called.

3. Inserts a row into `trial_results` with `cage_id`, `trial_id`, `outcome`, the event log, `session_id`, and `substage_id`.

4. If `session_id` is not null (i.e. this is a proper tracked run, not a dev one-shot), it looks up `subject_id` from the session row, then calls the advancement evaluator.

---

### Step 7 ‚Äî Advancement evaluation

`advancement.evaluate()` is called with `subject_id`, `substage_id`, and the open DB connection. It reads `advance_criteria` and `fallback_criteria` from `training_substages`. For each criterion it calls `_pct_correct()`:

- Queries `trial_results` for the last `window` non-aborted trials where `substage_id` matches AND the trial's session belongs to this subject.
- If fewer than `window` trials exist, returns False immediately ‚Äî not enough data.
- Otherwise counts how many are `correct` and divides by `window`.
- Compares against `threshold`.

Advance is checked first. If advance criteria are met, returns `"advance"`. If not, fallback is checked. If neither is met, returns `"stay"`.

---

### Step 8a ‚Äî Stay (criteria not met)

The runner is already in its ITI wait (from step 6.1). When the ITI expires, the runner loops back to step 3 and sends the next trial.

---

### Step 8b ‚Äî Advance or fallback (criteria met)

`advancement.apply()` is called:
- Reads `advance_to_substage_id` or `fallback_to_substage_id` from `training_substages`
- Updates `subjects.current_substage_id` to the new substage ID

Back in `handle_trial_event`, `stop_run()` is called for this cage. This sets the stop flag and fires the threading event, which interrupts the runner's ITI wait. The runner checks the stop flag, sees it is set, and exits the loop.

The next time the researcher opens a session for this animal, `subjects.current_substage_id` already points to the new substage, so the session automatically snapshots the new level.

---

### Manual stop

At any point the researcher can click **Trial ‚ñÝ**. The browser sends `POST /cage/N/trial/run/stop`. Flask sends `STOP_TRIAL` directly to the Pi over TCP (aborting any running trial immediately) and calls `stop_run()` on the runner. The runner's ITI wait is interrupted and the loop exits.

---

### Notes

- **Click expansion happens per-trial**: Rate parameters are stored compactly in the DB. Arrays are generated fresh in the runner on each iteration with `seed=None`, so every trial has a unique Poisson pattern even across repetitions.
- **The ITI is fully interruptible**: Both manual stop and automatic advancement interrupt the ITI immediately via `threading.Event.wait()`.
- **Advancement is synchronous with DB write**: The evaluator runs in the same connection after the result insert, so it always sees the just-written trial.
- **Dev one-shot path**: `POST /cage/N/trial/start` in `control.py` sends raw JSON directly to the Pi, bypassing the runner, session tracking, and advancement entirely. For hardware testing only.

## 4. UI Backend ‚Äî Core Modules

### `ui/runner.py` ‚Äî Continuous trial runner

Manages one background thread per cage. The thread loops indefinitely, sending trials to the Pi and waiting for results. There is no fixed repetition count ‚Äî the loop runs until `stop_run()` is called, either manually by the researcher or automatically by the advancement evaluator.

**State dict** (one entry per cage, keyed by `cage_id`):

- `thread` ‚Äî the daemon thread running `_run_loop`
- `event` ‚Äî a `threading.Event` used for two purposes: unblocked by `on_trial_complete()` when a result arrives, and fired by `stop_run()` to interrupt ITI waits
- `last_result` ‚Äî the most recent result dict received from the Pi
- `stop` ‚Äî boolean flag set by `stop_run()`; checked at the top of each loop iteration and after each ITI wait
- `session_id` ‚Äî stored at start, stamped on every trial result row
- `substage_id` ‚Äî stored at start, stamped on every trial result row

**Public API:**

| Function | Description |
|---|---|
| `start_run(cage_id, trial_definition, sender, base_iti_s, fail_iti_s, session_id, substage_id)` | Start a continuous run. Returns `(ok, msg)`. Fails if a run is already active on this cage. |
| `stop_run(cage_id)` | Signal the runner to stop. Sets `stop=True` and fires the event to interrupt any active ITI wait. |
| `on_trial_complete(cage_id, event)` | Called by `handle_trial_event` when the Pi result arrives. Stores result and fires the event to unblock the runner. |
| `get_run_context(cage_id)` | Returns `{session_id, substage_id}` for the active run. Used by `handle_trial_event` to stamp result rows. |
| `is_running(cage_id)` | Returns True if a runner thread is alive for this cage. |

**Loop steps per trial:**
1. Check stop flag ‚Äî exit if set
2. Call `_expand_clicks()` ‚Äî deep-copy the trial definition and replace any `play_clicks` rate parameters with freshly generated Poisson arrays
3. Serialise and send the trial JSON to the Pi over TCP
4. `event.wait(timeout=330s)` ‚Äî block until the result event is fired or the 330s watchdog timeout fires
5. Check stop flag again
6. Read the outcome from `last_result`; compute ITI (`base_iti_s` for correct, `fail_iti_s` for wrong/aborted)
7. `event.wait(timeout=iti)` ‚Äî interruptible ITI sleep; if the event fires during the wait, check the stop flag and exit if set

---

### `ui/advancement.py` ‚Äî Substage advancement evaluator

Pure functions with no Flask dependency. Called from `handle_trial_event` in `trial.py` after each trial result is written to the database.

**`evaluate(subject_id, substage_id, conn) ‚Üí "advance" | "fallback" | "stay"`**

Loads `advance_criteria`, `fallback_criteria`, `advance_to_substage_id`, and `fallback_to_substage_id` from `training_substages` for the given substage. Checks advance criteria first: if met and a target exists, returns `"advance"`. Then checks fallback criteria: if met and a target exists, returns `"fallback"`. If neither applies, returns `"stay"`. Null criteria always produce `"stay"`.

**`apply(subject_id, substage_id, decision, conn) ‚Üí new_substage_id | None`**

Reads the appropriate target ID from `training_substages` based on the decision, then executes `UPDATE subjects SET current_substage_id = new_id WHERE id = subject_id`. Returns the new substage ID for logging, or None if no target was defined.

**`_pct_correct(criteria, subject_id, substage_id, conn) ‚Üí bool`**

Implementation of the `pct_correct` criteria type. Queries `trial_results` for the last `window` trials where:
- `substage_id` matches the current substage
- `session_id` belongs to this subject (via subquery on `sessions`)
- `outcome` is either `correct` or `wrong` (aborted trials are excluded entirely)

Results are ordered by `completed_at DESC` so the most recent `window` trials are used. If fewer than `window` qualifying trials exist, returns False ‚Äî the animal has not yet done enough trials on this substage to be evaluated. Otherwise computes `correct_count / window` and compares to `threshold`.

> The window is strictly per-substage and per-subject. Trials from other substages or other animals are never counted. The effective window resets to zero each time an animal enters a new substage.

---

### `ui/click_generator.py` ‚Äî Poisson click train generator

**`generate_clicks(left_rate, right_rate, duration, seed=None) ‚Üí dict`**

Generates independent left and right Poisson click trains for one trial. Returns a dict with two keys:

- `left_clicks` ‚Äî sorted list of click times in seconds from trial start
- `right_clicks` ‚Äî sorted list of click times in seconds from trial start

Parameters:
- `left_rate` / `right_rate` ‚Äî mean click rates in clicks per second for each ear
- `duration` ‚Äî stimulus duration in seconds; no clicks are generated after this time
- `seed` ‚Äî optional integer RNG seed for reproducibility. When `None` (the default in the runner), NumPy uses OS entropy, so every call produces a unique pattern.

Inter-click intervals are drawn from an exponential distribution with mean `1/rate`, implementing a Poisson process. Difficulty is controlled by the ratio of `left_rate` to `right_rate` ‚Äî equal rates are the hardest (50/50 evidence), while a large ratio is easy (strong directional evidence). This follows the Poisson clicks task from Brunton et al. (2013).

Click arrays are generated fresh per trial inside `_expand_clicks()` in the runner. The database stores only the compact rate parameters (`left_rate`, `right_rate`, `click_duration`) in `task_config` ‚Äî the arrays themselves are never stored.
