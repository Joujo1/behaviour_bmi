## 1. Database Schema

### Tables and columns

There are six tables. All primary keys are auto-incrementing integers (`SERIAL`).

---

**`training_stages`**
Top-level groupings of the curriculum, e.g. "Habituation" or "Easy clicks".

- `id` — primary key
- `name` — unique text label shown in the UI
- `description` — optional free text
- `sort_order` — integer controlling display order in the curriculum editor sidebar

---

**`training_substages`**
One row per curriculum level. This is the central definition table — it carries both the trial definition (what the Pi runs) and the rules for when to move to the next level.

- `id` — primary key
- `stage_id` — foreign key → `training_stages.id`
- `substage_number` — integer ordering within the parent stage
- `label` — display name, e.g. "Easy left-biased"
- `task_config` — JSONB: the complete trial definition sent to the Pi (states, transitions, actions). Same format as what the curriculum editor builds.
- `advance_criteria` — JSONB: rules for moving forward. Null means manual-only advancement. Format: `{"type": "pct_correct", "window": 20, "threshold": 0.80}`
- `fallback_criteria` — JSONB: rules for moving backward. Same format. Null means no automatic fallback.
- `advance_to_substage_id` — foreign key → `training_substages.id` (self-referential). Null means end of curriculum.
- `fallback_to_substage_id` — foreign key → `training_substages.id` (self-referential). Null means no fallback defined.
- `retired` — boolean: if true, the substage is hidden from all UI dropdowns but not deleted

---

**`subjects`**
One row per animal.

- `id` — primary key
- `code` — unique text identifier, e.g. "R001"
- `sex` — single character: "M" or "F"
- `dob` — date of birth
- `weight_g` — body weight in grams
- `water_restricted` — boolean
- `enrolled_at` — timestamp when the animal was registered
- `current_substage_id` — foreign key → `training_substages.id`. This is updated automatically by the advancement evaluator after each trial, or manually by the researcher via the subjects page.
- `notes` — free text

---

**`sessions`**
One row per experimental sitting — one animal in one cage for one day's work.

- `id` — primary key
- `researcher` — name of who ran the session
- `notes` — free text
- `subject_id` — foreign key → `subjects.id`
- `substage_id` — foreign key → `training_substages.id`. Snapshotted from `subjects.current_substage_id` at the moment the session is opened. This means the session record stays correct even if the animal advances to a new substage mid-session.
- `weight_g` — animal weight recorded at session start or close
- `water_ml` — water given during the session
- `started_at` — timestamp when the session was opened
- `closed_at` — timestamp when the session was closed. Null means the session is still open.

---

**`trial_results`**
One row per trial completion or abort received from the Pi.

- `id` — primary key
- `cage_id` — integer: physical cage number (1–12)
- `trial_id` — text UUID generated by the Pi engine for this trial
- `outcome` — one of: `correct`, `wrong`, `aborted`
- `events` — JSONB array: the full timestamped event log from the Pi for this trial. Each entry has a time offset `t` in seconds from trial start, plus either a state transition (`from`/`to`) or a sensor event (`sensor`, `active`).
- `session_id` — foreign key → `sessions.id`. Null for one-shot dev runs that bypass the session workflow.
- `substage_id` — foreign key → `training_substages.id`. Null for one-shot dev runs. Used by the advancement evaluator to count only trials on the current substage.
- `completed_at` — timestamp when the result was received and written

---

**`recordings`**
Written by the frame writer in chunks of N frames (configured in `config.DB_CHUNK_SIZE`). Each row covers one chunk of a binary recording file for one cage. Chunked writes make the system crash-safe — partial recordings are still usable.

- `id` — bigserial primary key
- `cage_id` — physical cage number
- `chunk_start_frame` / `chunk_end_frame` — Pi sequence numbers for the first and last frame in this chunk
- `chunk_start_ts` / `chunk_end_ts` — microsecond timestamps from the Pi hardware clock
- `chunk_byte_offset` — byte position of this chunk's start in the binary file
- `chunk_frame_count` — number of frames in this chunk
- `created_at` — timestamp when this row was written

---

### Foreign key relationships

- `training_substages.stage_id` → `training_stages.id` (many substages belong to one stage)
- `training_substages.advance_to_substage_id` → `training_substages.id` (self-referential: points to the next level)
- `training_substages.fallback_to_substage_id` → `training_substages.id` (self-referential: points to the previous level)
- `subjects.current_substage_id` → `training_substages.id` (each animal is on exactly one substage at a time)
- `sessions.subject_id` → `subjects.id` (each session belongs to one animal)
- `sessions.substage_id` → `training_substages.id` (snapshotted at open time)
- `trial_results.session_id` → `sessions.id` (each trial belongs to one session)
- `trial_results.substage_id` → `training_substages.id` (each trial is tagged with the substage it was run on)

---

### Key design decisions

| Decision | Reason |
|---|---|
| `sessions.substage_id` is snapshotted at open time | Animal may advance mid-session; the session record stays accurate regardless |
| `trial_results.substage_id` is stamped per-trial | Advancement evaluator queries `WHERE substage_id = X` — only counts trials on the current substage, never trials from before the animal was on this level |
| `task_config` lives directly on `training_substages` | A substage IS a trial definition plus curriculum metadata; splitting them into separate tables adds complexity for no benefit |
| `advance_criteria` and `fallback_criteria` are JSONB | New criteria types can be added without schema changes |
| `recordings` tracks chunks not whole files | Crash-safe; partial recordings are still usable for analysis |

---

### Criteria JSONB format

```json
{
  "type":      "pct_correct",
  "window":    20,
  "threshold": 0.80
}
```

- `type` — criteria function to apply. Currently only `pct_correct` is implemented.
- `window` — how many recent trials to look back. Aborted trials are excluded and do not count toward the window.
- `threshold` — fraction of correct outcomes required, as a decimal between 0 and 1. In the UI this is entered as a percentage (0–100).

---

### `task_config` / trial definition format

This JSONB field is sent verbatim to the Raspberry Pi engine (after click arrays are expanded at runtime). Example:

```json
{
  "trial_id":      "trial_001",
  "initial_state": "cue_center",
  "states": [
    {
      "id":       "cue_center",
      "duration": 10.0,
      "entry_actions": [
        { "type": "led_on",      "target": "center" },
        { "type": "play_clicks", "left_rate": 39, "right_rate": 9, "click_duration": 1.0 }
      ],
      "exit_actions": [
        { "type": "led_off", "target": "center" }
      ],
      "transitions": [
        { "trigger": "beam_break", "target": "center", "hold_ms": 50, "next_state": "reward" },
        { "trigger": "timeout",                                        "next_state": "__wrong__" }
      ]
    },
    {
      "id":       "reward",
      "duration": 0.5,
      "entry_actions": [
        { "type": "valve_open", "target": "center" }
      ],
      "exit_actions": [
        { "type": "valve_close", "target": "center" }
      ],
      "transitions": [
        { "trigger": "timeout", "next_state": "__correct__" }
      ]
    }
  ]
}
```

Terminal state names with special meaning: `__correct__` (trial ends as correct), `__wrong__` (trial ends as wrong), `__end__` (trial ends as correct, legacy alias).

## 2. Full Setup — From Zero to Running Trials

This section describes how to go from a completely empty database to having trials running on a live animal. There are three one-time setup steps (curriculum, then subjects), and then a daily workflow that repeats every session.

---

### Step A — Build the curriculum (one time, before any animals)

The curriculum is the training ladder every animal climbs. It is made up of stages and substages. Stages are broad groupings (e.g. "Habituation", "Easy clicks", "Hard clicks"). Substages are the actual levels — each one defines exactly what trial the Pi runs and when an animal moves up or down.

**Navigate to `/curriculum` in the browser (linked from the dashboard header).**

#### A1. Create a stage

In the left sidebar, fill in a stage name and click **+ Stage**. A stage is just a label and an ordering. It appears immediately in the sidebar tree.

#### A2. Create substages inside the stage

Click on the stage name in the sidebar to expand it. Fill in a substage label and click **+ Substage**. A new empty substage appears under the stage. Click it to open the editor panel on the right.

#### A3. Build the trial definition

The editor panel has a state builder. This is where you define what the Pi does during each trial:

1. Type a state name in the **State** field and click **+ State** to create a state.
2. Select the state from the list — the action and transition editors below become active for that state.
3. Add **entry actions** (what happens when the state is entered — e.g. LED on, valve open, click playback).
4. Add **exit actions** (what happens when the state is left — e.g. LED off, valve close). These always fire regardless of which transition fires.
5. Add **transitions** — each transition has a trigger type (`beam_break`, `timeout`, `clicks_done`) and a target state. For `beam_break` you can also specify which beam and a `hold_ms` debounce time. Terminal states are `__correct__`, `__wrong__`, `__end__`.
6. Set the **duration** on the state if it needs a timeout transition.

The Graphviz diagram updates live as you build — it shows the full state machine with all transitions and action labels. Use this to visually verify the logic.

Click **Save trial definition** when done. This writes the `task_config` JSON to the substage row in the database.

#### A4. Set advancement criteria

Below the state builder, the **Criteria** bar has two rows: Advance and Fallback.

- **Advance**: select `pct_correct`, set a window (number of non-aborted trials to look back) and a threshold percentage. Select the substage the animal should move to when criteria are met.
- **Fallback**: same, but the substage the animal drops back to.

Both are optional. If left empty, advancement on that direction is manual only.

Click **Save criteria** to persist. This writes `advance_criteria`, `fallback_criteria`, `advance_to_substage_id`, and `fallback_to_substage_id` to the substage row.

#### A5. Repeat for all substages in the curriculum

Build every substage the animals will need. You can revisit and edit any substage at any time — click it in the sidebar to reload the editor with all saved values. Retired substages can be flagged with the **Retired** checkbox; they disappear from all dropdowns but are not deleted.

---

### Step B — Register subjects (one time per animal)

**Navigate to `/subjects-page` in the browser (linked from the dashboard header as "Subjects").**

Fill in the create form at the top:

- **Code** (required) — unique identifier, e.g. `R001`. This is the primary label used everywhere.
- **Sex** — M or F, optional.
- **DOB** — date of birth, optional.
- **Weight** — body weight in grams, optional.
- **Water restricted** — Yes/No.
- **Starting substage** — pick the substage the animal should begin on. This sets `subjects.current_substage_id` immediately. If left empty, the animal has no substage assigned and cannot be run until one is assigned.
- **Notes** — free text, optional.

Click **+ Add Subject**. The animal appears in the table below with its current stage and substage. The table is the ground truth for every animal's current position in the curriculum at any point in time.

**Manual substage override** — if an animal needs to be moved manually at any time (e.g. the researcher decides the animal is not ready), click **Move substage** on the animal's row. A modal opens with a full substage dropdown. Select the target and click **Move**. This updates `subjects.current_substage_id` directly.

---

### Step C — Daily session workflow (every training day)

This is the workflow the researcher repeats every day for every active animal.

#### C1. Open a session on the dashboard

**Navigate to `/` (the dashboard).**

Each active cage has a card. At the top of each card, use the subject dropdown to select the animal being run in that cage today. The dropdown shows all registered subjects. Selecting an animal automatically displays the animal's current substage name next to the dropdown — this is what will be run.

Click **Open Session**. This sends `POST /session/open` and does three things:
1. Creates a new row in `sessions` with `subject_id`, `researcher`, current timestamp as `started_at`.
2. Reads `subjects.current_substage_id` and copies it into `sessions.substage_id` as a snapshot.
3. Returns a `session_id` which is stored in the browser (`currentSessionId`).

The cage card now shows a green "Session open" status badge with the animal code and substage name. The trial start button becomes active.

#### C2. Start trials

Click **Trial ▶** on the cage card. The browser sends `POST /cage/N/trial/run` with the substage ID, session ID, and ITI values. Flask loads the `task_config` from the substage, starts a background runner thread, and immediately returns. The runner begins sending trials to the Pi continuously.

From this point, trials run automatically one after another without researcher input. The runner does not have a fixed trial count — it loops indefinitely. The researcher does not need to stay at the computer.

After each trial result arrives from the Pi, the advancement evaluator runs silently in the background. If the animal meets the advancement or fallback criteria, the runner stops automatically and `subjects.current_substage_id` is updated in the database. The researcher will see the runner status change to idle on next page refresh.

#### C3. Stop trials manually

Click **Trial ■** to stop at any time. This sends an immediate `STOP_TRIAL` to the Pi (aborting any in-progress trial) and stops the runner. If the animal is mid-ITI, the ITI is interrupted immediately.

#### C4. Close the session

At the end of the day's work, click **Close Session** on the cage card. This sends `POST /session/N/close`, which stamps `closed_at` on the session row. Optionally the researcher can record end-of-day weight and water given at this point.

Closing a session is important for data hygiene but does not affect the runner or the animal's substage. The runner can be stopped and sessions closed in any order.

#### C5. Multiple animals simultaneously

All of the above works for up to 12 animals in parallel. Each cage card is independent. Each cage has its own runner thread, its own TCP connection to the Pi, and its own session ID. There is no shared state between cages. A researcher can open sessions on all 12 cages, start all 12 runners, and let them all run concurrently.

---

### State of the database after a full day

After a typical training day:

- `sessions` — one new row per animal run today, `closed_at` set.
- `trial_results` — many new rows, each stamped with `session_id` and `substage_id`.
- `subjects.current_substage_id` — updated for any animal that met advancement or fallback criteria during the day.
- `recordings` — new chunks written for any cages that were streaming video.

The subjects page will show updated stage/substage positions for any animals that advanced. The curriculum editor shows no changes (the curriculum itself is unchanged; only where each animal sits within it has shifted).

## 3. Trial Data Flow

This describes what happens step by step from the researcher clicking **Trial ▶** through to automatic advancement firing.

### Preconditions
- A session has been opened for an animal (`POST /session/open`), which has recorded the animal's current substage.
- The dashboard has the session ID stored in `currentSessionId`.
- The runner dropdown is pre-populated with the animal's current substage.

---

### Step 1 — Researcher triggers the run

The researcher clicks **Trial ▶** on a cage card in the dashboard. The browser sends `POST /cage/N/trial/run` to Flask with:
- `substage_id` — which substage to run (pre-selected from the animal's current substage)
- `session_id` — the open session
- `base_iti_s` — inter-trial interval after a correct outcome
- `fail_iti_s` — inter-trial interval after a wrong or aborted outcome

---

### Step 2 — Flask loads the trial definition

Flask queries `training_substages` for the row with the given `substage_id` and reads `task_config`. This is the full trial JSON (states, transitions, actions). It then calls `start_run()` in `runner.py`, passing the trial definition, sender object, ITI values, session ID and substage ID. A background runner thread is started for the cage and Flask returns `{ok: true}` to the browser immediately.

---

### Step 3 — Runner loop begins

The runner thread enters a `while True` loop. At the top of each iteration:

1. The trial definition is deep-copied and `_expand_clicks()` is called. Any `play_clicks` action that stores rate parameters (`left_rate`, `right_rate`, `click_duration`) has those replaced with pre-generated Poisson click time arrays. A new random seed is used each time, so every trial has a unique click pattern.

2. The expanded trial JSON is serialised and sent to the Pi over TCP as a newline-terminated message.

3. The Pi responds with `ACK:ok` to confirm receipt.

---

### Step 4 — Pi runs the state machine

The Pi's `Engine` receives the trial JSON, loads the state machine, and enters the initial state. It drives hardware in response to events:
- Entry actions execute immediately on state entry (LEDs on, valves open, click playback starts)
- GPIO interrupts from beam sensors trigger beam_break transitions
- Timers fire timeout transitions when a state's duration expires
- The click watcher thread fires `clicks_done` transitions when both click trains finish naturally

The engine records every state transition and sensor event with a timestamp offset from trial start.

---

### Step 5 — Trial result returns to Flask

When the trial ends (either through a terminal state or the watchdog timer), the Pi's `on_trial_complete` callback sends a JSON event back to the server over TCP:

```json
{
  "event":    "trial_complete",
  "trial_id": "abc123",
  "outcome":  "correct",
  "events":   [{"t": 0.45, "from": "cue_center", "to": "reward"}, ...]
}
```

This arrives at `handle_trial_event()` in `trial.py`, which is called from the TCP reader thread.

---

### Step 6 — Result is processed

`handle_trial_event` does four things in order:

1. Calls `on_trial_complete()` in the runner — this sets the threading event, which unblocks the runner thread from its `event.wait()`. The runner is now free to start its ITI wait.

2. Reads `session_id` and `substage_id` from `get_run_context()` — these were stored when `start_run()` was called.

3. Inserts a row into `trial_results` with `cage_id`, `trial_id`, `outcome`, the event log, `session_id`, and `substage_id`.

4. If `session_id` is not null (i.e. this is a proper tracked run, not a dev one-shot), it looks up `subject_id` from the session row, then calls the advancement evaluator.

---

### Step 7 — Advancement evaluation

`advancement.evaluate()` is called with `subject_id`, `substage_id`, and the open DB connection. It reads `advance_criteria` and `fallback_criteria` from `training_substages`. For each criterion it calls `_pct_correct()`:

- Queries `trial_results` for the last `window` non-aborted trials where `substage_id` matches AND the trial's session belongs to this subject.
- If fewer than `window` trials exist, returns False immediately — not enough data.
- Otherwise counts how many are `correct` and divides by `window`.
- Compares against `threshold`.

Advance is checked first. If advance criteria are met, returns `"advance"`. If not, fallback is checked. If neither is met, returns `"stay"`.

---

### Step 8a — Stay (criteria not met)

The runner is already in its ITI wait (from step 6.1). When the ITI expires, the runner loops back to step 3 and sends the next trial.

---

### Step 8b — Advance or fallback (criteria met)

`advancement.apply()` is called:
- Reads `advance_to_substage_id` or `fallback_to_substage_id` from `training_substages`
- Updates `subjects.current_substage_id` to the new substage ID

Back in `handle_trial_event`, `stop_run()` is called for this cage. This sets the stop flag and fires the threading event, which interrupts the runner's ITI wait. The runner checks the stop flag, sees it is set, and exits the loop.

The next time the researcher opens a session for this animal, `subjects.current_substage_id` already points to the new substage, so the session automatically snapshots the new level.

---

### Manual stop

At any point the researcher can click **Trial ■**. The browser sends `POST /cage/N/trial/run/stop`. Flask sends `STOP_TRIAL` directly to the Pi over TCP (aborting any running trial immediately) and calls `stop_run()` on the runner. The runner's ITI wait is interrupted and the loop exits.

---

### Notes

- **Click expansion happens per-trial**: Rate parameters are stored compactly in the DB. Arrays are generated fresh in the runner on each iteration with `seed=None`, so every trial has a unique Poisson pattern even across repetitions.
- **The ITI is fully interruptible**: Both manual stop and automatic advancement interrupt the ITI immediately via `threading.Event.wait()`.
- **Advancement is synchronous with DB write**: The evaluator runs in the same connection after the result insert, so it always sees the just-written trial.
- **Dev one-shot path**: `POST /cage/N/trial/start` in `control.py` sends raw JSON directly to the Pi, bypassing the runner, session tracking, and advancement entirely. For hardware testing only.

## 4. UI Backend — Endpoints

All endpoints are Flask blueprints registered in `ui/ui_main.py`.

### `ui/endpoints/trial.py` — Trial execution and metrics

| Method | Path | Description |
|---|---|---|
| `GET` | `/metrics` | Per-cage aggregate stats from `trial_results`. Returns total, correct, wrong, aborted counts, success %, average response times. |
| `POST` | `/cage/<id>/trial/run` | Start a continuous run on a cage. Loads `task_config` from `training_substages`. Body: `substage_id`, `session_id`, `base_iti_s`, `fail_iti_s`. |
| `POST` | `/cage/<id>/trial/run/stop` | Stop the runner and send `STOP_TRIAL` to the Pi immediately. |

**Internal function `handle_trial_event(cage_id, event)`**

Called from `ui_main._on_pi_event()` when the TCP reader receives a `trial_complete` or `trial_aborted` event from the Pi. This is not an HTTP endpoint — it is the bridge between the TCP layer and the database/advancement system.

Sequence:
1. Calls `on_trial_complete()` to unblock the runner thread
2. Reads `session_id` and `substage_id` from `get_run_context()`
3. Inserts a row into `trial_results`
4. If `session_id` is set, looks up `subject_id` from `sessions`
5. Calls `advancement.evaluate()` and if non-stay, calls `advancement.apply()` then `stop_run()`

## 5. UI Backend — Core Modules

### `ui/runner.py` — Continuous trial runner

Manages one background thread per cage. The thread loops indefinitely, sending trials to the Pi and waiting for results. There is no fixed repetition count — the loop runs until `stop_run()` is called, either manually by the researcher or automatically by the advancement evaluator.

**State dict** (one entry per cage, keyed by `cage_id`):

- `thread` — the daemon thread running `_run_loop`
- `event` — a `threading.Event` used for two purposes: unblocked by `on_trial_complete()` when a result arrives, and fired by `stop_run()` to interrupt ITI waits
- `last_result` — the most recent result dict received from the Pi
- `stop` — boolean flag set by `stop_run()`; checked at the top of each loop iteration and after each ITI wait
- `session_id` — stored at start, stamped on every trial result row
- `substage_id` — stored at start, stamped on every trial result row

**Public API:**

| Function | Description |
|---|---|
| `start_run(cage_id, trial_definition, sender, base_iti_s, fail_iti_s, session_id, substage_id)` | Start a continuous run. Returns `(ok, msg)`. Fails if a run is already active on this cage. |
| `stop_run(cage_id)` | Signal the runner to stop. Sets `stop=True` and fires the event to interrupt any active ITI wait. |
| `on_trial_complete(cage_id, event)` | Called by `handle_trial_event` when the Pi result arrives. Stores result and fires the event to unblock the runner. |
| `get_run_context(cage_id)` | Returns `{session_id, substage_id}` for the active run. Used by `handle_trial_event` to stamp result rows. |
| `is_running(cage_id)` | Returns True if a runner thread is alive for this cage. |

**Loop steps per trial:**
1. Check stop flag — exit if set
2. Call `_expand_clicks()` — deep-copy the trial definition and replace any `play_clicks` rate parameters with freshly generated Poisson arrays
3. Serialise and send the trial JSON to the Pi over TCP
4. `event.wait(timeout=330s)` — block until the result event is fired or the 330s watchdog timeout fires
5. Check stop flag again
6. Read the outcome from `last_result`; compute ITI (`base_iti_s` for correct, `fail_iti_s` for wrong/aborted)
7. `event.wait(timeout=iti)` — interruptible ITI sleep; if the event fires during the wait, check the stop flag and exit if set

### `ui/advancement.py` — Substage advancement evaluator

Pure functions with no Flask dependency. Called from `handle_trial_event` in `trial.py` after each trial result is written to the database.

**`evaluate(subject_id, substage_id, conn) → "advance" | "fallback" | "stay"`**

Loads `advance_criteria`, `fallback_criteria`, `advance_to_substage_id`, and `fallback_to_substage_id` from `training_substages` for the given substage. Checks advance criteria first: if met and a target exists, returns `"advance"`. Then checks fallback criteria: if met and a target exists, returns `"fallback"`. If neither applies, returns `"stay"`. Null criteria always produce `"stay"`.

**`apply(subject_id, substage_id, decision, conn) → new_substage_id | None`**

Reads the appropriate target ID from `training_substages` based on the decision, then executes `UPDATE subjects SET current_substage_id = new_id WHERE id = subject_id`. Returns the new substage ID for logging, or None if no target was defined.

**`_pct_correct(criteria, subject_id, substage_id, conn) → bool`**

Implementation of the `pct_correct` criteria type. Queries `trial_results` for the last `window` trials where:
- `substage_id` matches the current substage
- `session_id` belongs to this subject (via subquery on `sessions`)
- `outcome` is either `correct` or `wrong` (aborted trials are excluded entirely)

Results are ordered by `completed_at DESC` so the most recent `window` trials are used. If fewer than `window` qualifying trials exist, returns False — the animal has not yet done enough trials on this substage to be evaluated. Otherwise computes `correct_count / window` and compares to `threshold`.

> The window is strictly per-substage and per-subject. Trials from other substages or other animals are never counted. The effective window resets to zero each time an animal enters a new substage.

### `ui/click_generator.py` — Poisson click train generator

**`generate_clicks(left_rate, right_rate, duration, seed=None) → dict`**

Generates independent left and right Poisson click trains for one trial. Returns a dict with two keys:

- `left_clicks` — sorted list of click times in seconds from trial start
- `right_clicks` — sorted list of click times in seconds from trial start

Parameters:
- `left_rate` / `right_rate` — mean click rates in clicks per second for each ear
- `duration` — stimulus duration in seconds; no clicks are generated after this time
- `seed` — optional integer RNG seed for reproducibility. When `None` (the default in the runner), NumPy uses OS entropy, so every call produces a unique pattern.

Inter-click intervals are drawn from an exponential distribution with mean `1/rate`, implementing a Poisson process. Difficulty is controlled by the ratio of `left_rate` to `right_rate` — equal rates are the hardest (50/50 evidence), while a large ratio is easy (strong directional evidence). This follows the Poisson clicks task from Brunton et al. (2013).

Click arrays are generated fresh per trial inside `_expand_clicks()` in the runner. The database stores only the compact rate parameters (`left_rate`, `right_rate`, `click_duration`) in `task_config` — the arrays themselves are never stored.
